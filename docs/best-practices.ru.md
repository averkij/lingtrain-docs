# Лучшие практики для качественного выравнивания {#best-practices}

В этом руководстве собраны экспертные советы и стратегии для достижения максимального качества выравнивания с помощью Lingtrain Aligner. Независимо от того, выравниваете ли вы роман, техническое руководство или коллекцию коротких текстов, эти рекомендации помогут избежать типичных ошибок и получить чистый результат, готовый к публикации.

## Подготовка текстов {#text-preparation}

Качественное выравнивание начинается с качественного ввода. Время, вложенное в подготовку текстов, напрямую окупается качеством выравнивания.

### Очистка текстов {#cleaning}

Перед загрузкой удалите содержимое, не имеющее аналога в другом языке:

- **Номера страниц** — всегда удаляйте; они присутствуют только в структуре одного текста
- **Колонтитулы** — повторяющийся текст из макета страниц
- **Маркеры сносок** — встроенные надстрочные номера, ссылающиеся на сноски
- **Сами сноски** — если только оба текста не имеют совпадающих сносок
- **Оглавление** — обычно структурировано по-разному в разных изданиях
- **Издательская информация** — авторские права, ISBN, данные об издании
- **Предисловия переводчика** — если только вы не хотите их специально выровнять

### Рекомендации по форматированию {#formatting}

- **Разделяйте абзацы пустыми строками.** Алгоритм разбиения использует границы абзацев для улучшения сегментации. Текст одним сплошным блоком без пустых строк даст худшее разбиение.
- **Используйте единообразную пунктуацию.** Смешение стилей пунктуации (например, чередование «...» и «…») может сбивать алгоритм.
- **Удалите лишние пробелы.** Множественные последовательные пробелы и табуляции нормализуйте до одинарных пробелов.
- **Исправьте проблемы кодировки до загрузки.** Типичные проблемы: «кракозябры» от несовпадения кодировок, умные кавычки вместо обычных, невидимые Unicode-символы.

### Добавление тегов разметки {#adding-tags}

Всегда добавляйте как минимум:

1. **`%%%%%author.`** — имя автора в обоих текстах
2. **`%%%%%title.`** — название произведения в обоих текстах
3. **`%%%%%h1.`** — заголовки глав в обоих текстах

Эти теги формируют титульную страницу и структуру глав в экспортированной HTML-книге. Без них книга будет плоской последовательностью предложений без визуальной структуры.

**Проверяйте согласованность тегов** между текстами:
- Одинаковое количество каждого типа тегов
- Теги идут в одном порядке
- Используйте панель Marks на вкладке Documents для проверки

### Работа с разными изданиями {#different-editions}

Если ваши тексты из разных изданий:

- **Сокращённые издания:** Удалите из полного текста фрагменты, отсутствующие в сокращённой версии. Выравнивание текстов с резко различающейся длиной приведёт к множеству конфликтов.
- **Различная абзацная структура:** Это нормально — алайнер справляется с разной структурой абзацев. Главное, чтобы содержание предложений совпадало.
- **Добавления/пропуски переводчика:** Если переводчик добавил пояснительные заметки или опустил отрывки, удалите их из одной стороны или смиритесь с конфликтами в соответствующем разделе.

## Выбор модели {#choosing-model}

Модель эмбеддингов — основа качества выравнивания. Выбирайте исходя из языковой пары:

| Сценарий | Рекомендуемая модель |
|----------|---------------------|
| Основные европейские пары (EN-FR, EN-DE, EN-ES и т. д.) | Модель по умолчанию или LaBSE |
| Русский + любой крупный язык | LaBSE или модель по умолчанию |
| Китайский или японский + любой язык | LaBSE |
| Малоресурсные языки (башкирский, чувашский и т. д.) | LaBSE + подстрочник |
| Крайне малоресурсные языки | SONAR + подстрочник |
| Русскоязычная задача (RU-UK, RU-BE) | LaBSE или rubert-tiny2 |
| Нужен максимальный языковой охват | SONAR |

**Если сомневаетесь, используйте LaBSE.** Она обеспечивает лучший баланс качества и языкового охвата для большинства случаев.

Модель фиксируется после первого запуска выравнивания. Чтобы попробовать другую модель, используйте функцию «Erase» для сброса выравнивания, затем выберите новую модель.

## Оптимизация размера батча и окна {#batch-optimization}

### Размер батча {#batch-size}

Размер батча по умолчанию (**200**) хорошо подходит для большинства текстов. Рассмотрите корректировку в этих случаях:

- **Очень короткие тексты** (менее 100 предложений) — текст поместится в один батч. Корректировка не нужна.
- **Очень длинные тексты** (10 000+ предложений) — размер 200 подходит. Он создаёт много батчей, но каждый обрабатывается быстро.

Примечание: размер батча фиксируется при создании выравнивания и не может быть изменён позже.

### Размер окна {#window-size}

Параметр «окно» добавляет дополнительные предложения по обе стороны пропорционального диапазона:

- **По умолчанию (40-50):** Подходит для текстов со схожим числом предложений на раздел
- **Увеличенное (60-80):** Используйте при значительном различии числа предложений (переводчик разделил многие предложения, или один текст значительно многословнее)
- **Очень большое (100+):** Требуется редко. Увеличивает время вычислений и риск ложных совпадений

### Параметр «сдвиг» {#shift-usage}

Сдвиг — ваш основной инструмент для коррекции дрейфа выравнивания:

1. **Начните со сдвига = 0** для первого батча
2. **Проверьте визуализацию** после первого батча
3. Если диагональ уходит вверх (целевые предложения «опережают»), используйте **положительный сдвиг**
4. Если диагональ уходит вниз, используйте **отрицательный сдвиг**
5. Корректируйте на 10-20 предложений за раз; проверяйте визуализацию после каждой корректировки
6. Разные батчи могут требовать разных сдвигов

## Когда использовать подстрочник {#when-proxy}

Используйте подстрочник, когда:

1. **Качество выравнивания плохое** — много конфликтов, нарушенная диагональ
2. **Один из языков малоресурсный** — мало обучающих примеров в модели эмбеддингов
3. **Языковая пара дистантная** — модель плохо справляется с межъязыковым сходством

**Как подготовить хороший подстрочник:**

1. Скачайте разбитый текст из предпросмотра предложений
2. Переведите его лучшим доступным сервисом МП для данного языка
3. Убедитесь, что число строк совпадает точно
4. Загрузите как прокси на странице деталей выравнивания

**Выбор языка подстрочника:** Переводите на язык, хорошо представленный в модели эмбеддингов. Английский и русский — безопасный выбор для большинства моделей.

## Работа с различными жанрами {#genres}

### Художественная литература (романы, рассказы) {#fiction}

- **Ожидайте некоторое количество конфликтов.** Литературный перевод включает творческие решения — объединение/разделение предложений, перестановку частей. Часть конфликтов неизбежна.
- **Обильно используйте теги разметки.** Добавляйте заголовки глав, разделители секций и эпиграфы.
- **Абзацная структура важна.** Включайте маркеры абзацев для лучшего вывода HTML-книги.
- **Диалоговые фрагменты** могут порождать больше коротких предложений; это нормально.

### Техническая документация {#technical}

- **Более высокая последовательность.** Технические переводы, как правило, более буквальны, что даёт лучшее выравнивание.
- **Следите за таблицами и списками.** Они часто нарушают разбиение на предложения. Извлеките их отдельно или удалите.
- **Блоки кода** следует удалить или последовательно сохранить в обоих текстах.

### Академические/научные статьи {#academic}

- **Удалите цитаты**, которые присутствуют только в формате одного текста.
- **Уравнения и формулы** могут вызывать проблемы с разбиением. Размещайте их на отдельных строках.
- **Аннотации** обычно хорошо выравниваются, так как переводятся точно.
- **Библиографию** следует удалить, если тексты не имеют совпадающих ссылок.

### Поэзия {#poetry}

- **Поэзия принципиально сложна** для выравнивания на основе предложений, поскольку разрывы строк являются структурными, а не синтаксическими.
- Рассмотрите возможность трактовать каждую строфу как единицу (разделённую пустыми строками), а не каждую строку.
- Ручное редактирование, скорее всего, потребуется.

## Стратегия для длинных текстов {#long-texts}

Для текстов с 5 000+ предложениями:

1. **Обрабатывайте поэтапно.** Выравнивайте по 3-5 батчей за раз, а не все сразу.
2. **Проверяйте визуализацию после каждого этапа.** Ловите дрейф рано.
3. **Корректируйте сдвиг постепенно.** Если диагональ дрейфует в батче 3, применяйте корректирующий сдвиг начиная с батча 4.
4. **Разрешайте конфликты после всех батчей.** Запустите автоматическое разрешение один раз, когда все батчи обработаны.
5. **Не перевыравнивайте завершённые батчи**, если нет явной проблемы.
6. **Экспортируйте резервные копии (.lt-файлы)** на ключевых этапах.

## Процесс проверки качества {#quality-verification}

После завершения выравнивания и разрешения конфликтов проверьте качество:

### Шаг 1: Проверьте общие метрики {#verify-metrics}

- **Chain score** должен быть выше 0.95 для хорошего качества
- **Побатчовое качество** должно быть последовательным; один батч с низким качеством может указывать на проблему в данной области

### Шаг 2: Выборочная проверка в редакторе {#verify-editor}

- Откройте редактор и перейдите к середине текста
- Проверьте 10-20 пар предложений на корректность
- Проверьте начало и конец текста (краевые случаи)
- Ищите закономерности: сконцентрированы ли ошибки в одной области?

### Шаг 3: Проверьте визуализацию {#verify-visualization}

- Все точечные графики батчей должны показывать чистую диагональ
- Ищите выбросы — точки далеко от диагонали указывают на конкретные несоответствия
- Ступенчатые диагонали допустимы (указывают на добавленные/удалённые блоки содержимого)

### Шаг 4: Предпросмотр книги {#verify-preview}

- Сгенерируйте предпросмотр книги на вкладке Create
- Проверьте структуру глав и разрывы абзацев
- Убедитесь, что теги разметки дали корректное форматирование
- Попробуйте оба варианта направления абзацев, чтобы увидеть, какой читается лучше

## Типичные ошибки {#common-pitfalls}

### Ошибка 1: Не проверять первый батч {#pitfall-first-batch}

**Проблема:** Обработка всех батчей сразу без проверки первого.

**Почему это важно:** Если первый батч имеет нарушенную диагональ, все последующие батчи будут прогрессивно хуже, поскольку позиции окон основаны на пропорции.

**Решение:** Всегда выравнивайте один батч первым, проверяйте визуализацию, корректируйте параметры при необходимости, затем продолжайте.

### Ошибка 2: Игнорировать несовпадение разметки {#pitfall-markup}

**Проблема:** В одном тексте есть теги разметки, которых нет в другом.

**Почему это важно:** Несовпадающие теги вызывают структурные нарушения в экспортированной книге. Отсутствие тегов означает отсутствие заголовков глав и титульной страницы.

**Решение:** Всегда проверяйте панель Marks после загрузки. Исправьте несоответствия до создания выравнивания.

### Ошибка 3: Неосторожное использование «Align All» {#pitfall-align-all}

**Проблема:** Использование «Align all» после ручных правок.

**Почему это важно:** «Align all» стирает все предыдущие результаты, включая ручные правки, разрешённые конфликты и пользовательские значения сдвига.

**Решение:** Используйте выборочное перевыравнивание (конкретные ID батчей). Экспортируйте резервную копию перед любым крупным перевыравниванием.

### Ошибка 4: Пропускать автоматическое разрешение {#pitfall-skip-resolution}

**Проблема:** Попытка исправить все конфликты вручную в редакторе.

**Почему это важно:** Автоматическое разрешение корректно обрабатывает 70-90% конфликтов и значительно быстрее ручного редактирования.

**Решение:** Всегда запускайте автоматическое разрешение первым. Вручную исправляйте только конфликты, оставшиеся после автоматического разрешения.

### Ошибка 5: Неправильные версии текстов {#pitfall-wrong-version}

**Проблема:** Выравнивание текстов из разных изданий, сокращённых версий или значительно модифицированных текстов.

**Почему это важно:** Если тексты фактически не соответствуют друг другу, никакая настройка параметров не даст хорошего выравнивания.

**Решение:** Убедитесь, что тексты являются настоящими переводами друг друга. Сравните объём слов — соотношение должно быть разумным (обычно 0.7-1.4 для большинства языковых пар).

### Ошибка 6: Забыть подстрочник для малоресурсных языков {#pitfall-no-proxy}

**Проблема:** Попытка прямого выравнивания малоресурсного языка без подстрочника.

**Почему это важно:** Модель эмбеддингов может выдавать бессмысленные векторы для недостаточно представленных языков, что приводит к почти случайному выравниванию.

**Решение:** Проверьте поддержку языковой пары. Если один язык плохо представлен — создайте подстрочник через машинный перевод.

## Итоговый чек-лист {#checklist}

Перед началом выравнивания:

- [ ] Тексты в формате .txt, кодировка UTF-8
- [ ] Удалён посторонний контент (номера страниц, сноски, оглавление)
- [ ] Абзацы разделены пустыми строками
- [ ] Добавлены теги разметки (минимум: author, title, заголовки глав)
- [ ] Панель Marks показывает совпадающее количество тегов
- [ ] Выбрана подходящая модель эмбеддингов для языковой пары
- [ ] Подготовлен подстрочник, если нужен для малоресурсных языков

Во время выравнивания:

- [ ] Первый батч проверен через визуализацию перед продолжением
- [ ] Сдвиг скорректирован при дрейфе диагонали
- [ ] Батчи обработаны управляемыми группами (3-5 за раз)
- [ ] Автоматическое разрешение конфликтов запущено после всех батчей

После выравнивания:

- [ ] Chain score выше 0.95
- [ ] Выборочная проверка в редакторе (начало, середина, конец)
- [ ] Предпросмотр книги проверен на структуру и форматирование
- [ ] Экспортирована резервная копия (.lt-файл)
