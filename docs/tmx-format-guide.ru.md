# Руководство по формату TMX {#tmx}

Translation Memory eXchange (TMX) — это отраслевой стандарт обмена данными памяти переводов между различными инструментами и системами. Lingtrain Aligner может экспортировать выровненные пары предложений в формате TMX, что делает результаты выравнивания непосредственно пригодными для использования в профессиональных переводческих процессах. В этом руководстве рассматривается спецификация формата TMX, его XML-структура и способы использования TMX-файлов с популярными CAT-инструментами.

## Что такое TMX? {#what-is-tmx}

TMX — это формат на основе XML, предназначенный для представления данных памяти переводов — пар (или групп) переведённых сегментов на двух или более языках. Он был разработан LISA (Localization Industry Standards Association) и ныне поддерживается GALA (Globalization and Localization Association). Текущая широко используемая версия — TMX 1.4b, опубликованная в 2005 году.

Ключевые принципы TMX:

- **Независимость от инструментов**: TMX-файлы можно импортировать в любой совместимый переводческий инструмент
- **Языковая нейтральность**: Любое количество языков может быть представлено в одном TMX-файле
- **Богатство метаданных**: Каждый сегмент может нести метаданные о своём происхождении, дате создания, переводчике и качестве
- **Поддержка Unicode**: Полная поддержка Unicode для любой системы письма

## XML-структура {#structure}

TMX-файл — это XML-документ с чётко определённой структурой:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE tmx SYSTEM "tmx14.dtd">
<tmx version="1.4">
  <header
    creationtool="Lingtrain Aligner"
    creationtoolversion="1.0"
    segtype="sentence"
    o-tmf="text"
    adminlang="en"
    srclang="en"
    datatype="plaintext"
  />
  <body>
    <tu>
      <tuv xml:lang="en">
        <seg>The weather was beautiful that morning.</seg>
      </tuv>
      <tuv xml:lang="ru">
        <seg>Погода в то утро была прекрасной.</seg>
      </tuv>
    </tu>
    <tu>
      <tuv xml:lang="en">
        <seg>She decided to go for a walk in the park.</seg>
      </tuv>
      <tuv xml:lang="ru">
        <seg>Она решила пойти погулять в парке.</seg>
      </tuv>
    </tu>
  </body>
</tmx>
```

### Основные элементы {#key-elements}

- **`<tmx>`** — Корневой элемент, указывает версию TMX
- **`<header>`** — Содержит метаданные о файле: какой инструмент его создал, исходный язык, тип сегментации и формат данных
- **`<body>`** — Содержит все единицы перевода
- **`<tu>`** — Translation Unit (единица перевода), одна группа выровненных сегментов. Каждый `<tu>` содержит один или более элементов `<tuv>`
- **`<tuv>`** — Translation Unit Variant (вариант единицы перевода), одна языковая версия сегмента. Атрибут `xml:lang` указывает код языка
- **`<seg>`** — Собственно текстовый сегмент

### Атрибуты заголовка {#header-attributes}

Элемент `<header>` поддерживает ряд атрибутов:

| Атрибут | Обязателен | Описание |
|---------|-----------|----------|
| `creationtool` | Да | Название инструмента, создавшего файл |
| `creationtoolversion` | Да | Версия инструмента |
| `segtype` | Да | Тип сегментации: `sentence`, `paragraph` или `block` |
| `o-tmf` | Да | Исходный формат памяти переводов |
| `adminlang` | Да | Язык для административных метаданных |
| `srclang` | Да | Код исходного языка (например, "en", "ru", "zh") |
| `datatype` | Да | Тип данных: `plaintext`, `html`, `xml` и т.д. |
| `creationdate` | Нет | Дата создания файла (формат ISO 8601) |
| `creationid` | Нет | Идентификатор создателя файла |

### Атрибуты единиц перевода {#tu-attributes}

Каждый `<tu>` может содержать дополнительные метаданные:

- `tuid` — Уникальный идентификатор единицы перевода
- `changedate` — Дата последнего изменения
- `changeid` — Кто последним изменил единицу
- `creationdate` — Когда единица была создана
- `creationid` — Кто создал единицу
- `srclang` — Исходный язык (переопределяет заголовок, если отличается)

### Коды языков {#language-codes}

TMX использует коды языков в соответствии с RFC 4646 (ныне RFC 5646). Распространённые коды:

| Код | Язык |
|-----|------|
| `en` | Английский |
| `en-US` | Английский (США) |
| `en-GB` | Английский (Великобритания) |
| `ru` | Русский |
| `de` | Немецкий |
| `fr` | Французский |
| `es` | Испанский |
| `zh` | Китайский |
| `zh-CN` | Китайский (упрощённый) |
| `zh-TW` | Китайский (традиционный) |
| `ja` | Японский |
| `ko` | Корейский |
| `ar` | Арабский |
| `pt-BR` | Португальский (Бразилия) |

## Как Lingtrain экспортирует TMX {#lingtrain-export}

При экспорте результатов выравнивания в формате TMX из Lingtrain:

1. Каждая цепочка выравнивания становится одним элементом `<tu>`
2. Многопредложенческие цепочки (например, 2:1) имеют предложения, объединённые через пробел в элементе `<seg>`
3. Заголовок включает Lingtrain как инструмент создания
4. Коды языков устанавливаются на основе языков, выбранных при настройке выравнивания
5. Тип сегментации устанавливается как `sentence`

Полученный TMX-файл непосредственно импортируется в любой TMX-совместимый CAT-инструмент.

## Импорт TMX в CAT-инструменты {#importing}

### SDL Trados Studio {#trados}

SDL Trados Studio (ныне Trados Studio) — наиболее широко используемый профессиональный CAT-инструмент.

Для импорта TMX-файла:

1. Откройте Trados Studio
2. Перейдите в раздел **Translation Memories**
3. Нажмите **New** для создания новой памяти переводов или выберите существующую
4. Щёлкните правой кнопкой на памяти и выберите **Import**
5. Укажите ваш TMX-файл и нажмите **Open**
6. В настройках импорта сконфигурируйте:
   - **Import action**: Выберите, добавлять новые записи, перезаписывать существующие или объединять
   - **Field mapping**: Сопоставьте пользовательские поля из TMX с полями Trados
7. Нажмите **Import** для начала процесса

Trados сообщит число импортированных записей, найденные дубликаты и обнаруженные ошибки.

### memoQ {#memoq}

memoQ — популярный CAT-инструмент, особенно на европейских рынках.

Для импорта TMX:

1. Откройте memoQ и перейдите в **Resource Console** (или **Translation memories** в настройках проекта)
2. Создайте новую память переводов или выберите существующую
3. Нажмите **Import/Export**, затем **Import from TMX**
4. Выберите ваш TMX-файл
5. Настройте параметры импорта:
   - **Handling of duplicates**: Добавить, пропустить или перезаписать
   - **Segment status**: Установите статус импортированных сегментов (подтверждён, не подтверждён)
6. Нажмите **Import**

memoQ обрабатывает TMX-файлы из Lingtrain без проблем, поскольку Lingtrain создаёт TMX, соответствующий стандарту.

### OmegaT {#omegat}

OmegaT — бесплатный CAT-инструмент с открытым исходным кодом, популярный в академической и некоммерческой среде.

Для использования TMX-файлов с OmegaT:

1. Поместите TMX-файл в папку **tm** вашего проекта OmegaT:
   - `project_root/tm/` для справочных памятей (нечёткие совпадения показываются, но не вставляются автоматически)
   - `project_root/tm/auto/` для автоматически вставляемых совпадений (100% совпадения вставляются автоматически)
   - `project_root/tm/enforce/` для обязательных совпадений (всегда используются при точном совпадении)
2. Перезагрузите проект (**Project > Reload**)
3. OmegaT автоматически прочитает TMX-файл и покажет совпадения в процессе перевода

OmegaT использует TMX как свой родной формат для памятей переводов, что делает импорт бесшовным.

### Memsource (Phrase TMS) {#memsource}

Memsource, ныне известный как Phrase TMS, — облачная система управления переводами.

Для импорта TMX:

1. Войдите в Phrase TMS
2. Перейдите в раздел **Translation Memories**
3. Нажмите **New** для создания памяти переводов или выберите существующую
4. Нажмите **Import** в детальном виде памяти
5. Загрузите TMX-файл
6. Настройте параметры импорта (обработка дубликатов, статус сегментов)
7. Нажмите **Import**

Phrase TMS быстро обрабатывает TMX-файлы, даже для больших файлов с сотнями тысяч сегментов.

### Другие инструменты {#other-tools}

TMX также поддерживается:

- **Wordfast** (Pro и Classic) — импорт через меню Translation Memory
- **MateCat** — загрузка TMX при настройке проекта
- **CafeTran Espresso** — перетащите TMX в панель памяти переводов
- **Déjà Vu** — импорт через меню Memory
- **Across** — импорт через управление crossTank

## Редактирование TMX-файлов {#editing}

TMX-файлы — это обычный XML, поэтому их можно редактировать в любом текстовом редакторе. Однако для больших файлов или сложных операций удобнее специализированные инструменты:

### Olifant {#olifant}

Olifant (часть Okapi Framework) — бесплатный редактор TMX, предоставляющий:
- Визуальный просмотр и поиск по содержимому TMX
- Фильтрацию по языку, текстовому содержимому или метаданным
- Редактирование отдельных сегментов
- Пакетные операции (поиск и замена, удаление записей по условию)
- Разбиение и объединение TMX-файлов

### TMXEditor от Maxprograms {#tmxeditor}

Кроссплатформенный редактор TMX, предлагающий:
- Валидацию по спецификации TMX
- Поиск и замену с поддержкой регулярных выражений
- Удаление дубликатов
- Фильтрацию и экспорт подмножеств

### Программное редактирование {#programmatic}

Для пакетной обработки TMX-файлы можно обрабатывать любой XML-библиотекой. Пример на Python:

```python
import xml.etree.ElementTree as ET

tree = ET.parse('alignment.tmx')
root = tree.getroot()

for tu in root.iter('tu'):
    for tuv in tu.findall('tuv'):
        lang = tuv.get('{http://www.w3.org/XML/1998/namespace}lang')
        seg = tuv.find('seg')
        print(f"[{lang}] {seg.text}")
```

## Объединение памятей переводов {#merging}

При работе над крупными проектами может потребоваться объединить TMX-файлы из разных источников (например, разные главы, выровненные по отдельности):

### Простая конкатенация {#concatenation}

Для базового объединения можно объединить элементы `<tu>` из нескольких TMX-файлов в один `<body>`. Убедитесь, что атрибуты `<header>` согласованы (одинаковые коды языков, одинаковый тип сегментации).

### Дедупликация {#deduplication}

После объединения могут появиться дубликаты (одна и та же пара «источник — цель» встречается несколько раз). Большинство CAT-инструментов обрабатывают это при импорте, но можно дедуплицировать заранее:

1. Импортируйте все TMX-файлы в одну память переводов в вашем CAT-инструменте
2. Экспортируйте объединённую память как новый TMX-файл
3. Процесс экспорта обычно удаляет точные дубликаты

Альтернативно используйте инструмент вроде Olifant для выявления и удаления дубликатов.

### Обработка конфликтов {#merge-conflicts}

Когда один и тот же исходный сегмент присутствует с разными переводами (от разных переводчиков или из разных версий текста), нужна стратегия объединения:

- **Сохранить все**: Сохранить все варианты. Большинство CAT-инструментов покажут самый последний как основное совпадение.
- **Сохранить последний**: Предпочесть самую недавно созданную запись.
- **Сохранить лучший по качеству**: Если у записей есть оценки качества, сохранить запись с наивысшей оценкой.

## Детали стандарта TMX 1.4 {#standard}

### Встроенные элементы {#inline}

TMX поддерживает встроенные элементы внутри `<seg>` для представления форматирования:

- `<bpt>` / `<ept>` — Начало/конец парных тегов (например, жирный, курсив)
- `<it>` — Изолированный тег
- `<ph>` — Заполнитель (например, ссылка на изображение)
- `<hi>` — Выделенный текст
- `<sub>` — Подпоток (встроенное содержимое)

Lingtrain экспортирует сегменты как обычный текст без встроенного форматирования. Если исходные тексты содержали форматирование, оно не сохранится в TMX-экспорте.

### Кодировка символов {#encoding}

TMX-файлы должны быть валидным XML и должны использовать кодировку UTF-8 (по умолчанию). Специальные символы должны быть правильно экранированы:

- `&` становится `&amp;`
- `<` становится `&lt;`
- `>` становится `&gt;`
- `"` становится `&quot;` (в атрибутах)

Lingtrain обрабатывает это экранирование автоматически при экспорте.

### Уровни совместимости {#compliance}

Стандарт TMX определяет два уровня совместимости:

- **Уровень 1**: Только текстовые сегменты (без встроенного форматирования). Это то, что производит Lingtrain.
- **Уровень 2**: Форматированный текст с встроенными тегами форматирования. Необходим для сохранения форматирования из CAT-инструментов.

TMX-файлы уровня 1 универсально совместимы. Файлы уровня 2 могут иметь специфичные для инструмента конвенции встроенных тегов, ограничивающие переносимость.

## Лучшие практики {#best-practices}

1. **Используйте единообразные коды языков**: Убедитесь, что ваши TMX-файлы используют одинаковые коды языков во всех проектах. Смешивание `en` и `en-US` может вызвать ошибки сопоставления в некоторых инструментах.

2. **Включайте метаданные**: Добавление даты создания и информации об инструменте помогает отслеживать происхождение записей памяти переводов.

3. **Валидируйте перед импортом**: Используйте валидатор TMX (например, встроенный в Maxprograms TMXEditor) для проверки структурных проблем перед импортом в рабочие памяти.

4. **Делайте резервные копии перед объединением**: Всегда создавайте резервные копии существующих памятей переводов перед импортом новых TMX-данных, на случай повреждения или непредвиденных перезаписей.

5. **Проверяйте качество выравнивания**: TMX-файлы из автоматического выравнивания (как Lingtrain) следует проверять на точность, прежде чем добавлять в рабочие памяти переводов. Записи низкого качества могут ухудшить переводческие подсказки.

6. **Сегментируйте правильно**: TMX-файлы лучше всего работают с сегментацией на уровне предложений. Сегментация на уровне абзацев даёт меньше, но менее переиспользуемых совпадений.
