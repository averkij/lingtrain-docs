# Алгоритм выравнивания {#algorithm}

На этой странице объясняется, как работает Lingtrain Aligner внутри — математика, модели и конвейер обработки, превращающий два необработанных текста в выровненные пары предложений.

## Обзор {#overview}

Конвейер выравнивания состоит из четырёх этапов:

1. **Предобработка** — разбиение текстов на предложения, извлечение метаданных
2. **Эмбеддинги** — преобразование предложений в векторные представления с помощью ML-моделей
3. **Сопоставление** — построение матрицы сходства и выбор лучших соответствий
4. **Разрешение конфликтов** — исправление несоответствий, где сопоставление «один к одному» не удалось

## Разбиение на предложения {#splitting}

Перед выравниванием каждый текст разбивается на отдельные предложения. Lingtrain использует языко-зависимые правила разбиения:

- **Русский, украинский, белорусский** — библиотека `razdel`, основанная на правилах и разработанная для кириллических текстов
- **Китайский** — регулярные выражения для знаков конца предложения (`。！？`)
- **Японский** — аналогичное сопоставление с японской пунктуацией
- **Другие языки** — стандартный токенизатор с языко-специфичными правилами

При разбиении теги разметки (`%%%%%author.`, `%%%%%h1.` и т.д.) извлекаются и сохраняются как метаданные. Строки с тегами исключаются из процесса выравнивания — они используются только для форматирования при экспорте.

Границы абзацев сохраняются как маркеры на последнем предложении каждого абзаца. Эти маркеры позднее используются для восстановления структуры абзацев в экспортированных книгах.

## Эмбеддинги предложений {#embeddings}

Каждое предложение преобразуется в плотный вектор (эмбеддинг), который отражает его семантическое значение. Lingtrain поддерживает несколько мультиязычных моделей эмбеддингов:

| Модель | Языки | Описание |
|--------|-------|----------|
| **distiluse-base-multilingual-cased-v2** | 50+ | Модель по умолчанию. Быстрая, хорошее качество для основных языковых пар. Основана на дистиллированном Universal Sentence Encoder. |
| **LaBSE** | 100+ | Language-agnostic BERT Sentence Embeddings. Лучший выбор для редких и малоресурсных языков. |
| **XLM-R-100langs** | 100+ | Модель на основе XLM-RoBERTa с широким мультиязычным покрытием. |
| **rubert-tiny2** | Русский | Легковесная модель Russian BERT. Быстрый инференс для задач с русским языком. |
| **SONAR** | 200+ | Массивный мультиязычный кодировщик от Meta. Самое широкое языковое покрытие. |

Эмбеддинги вычисляются пакетами (по умолчанию: 10 предложений за раз) и L2-нормализуются перед сохранением. Нормализованные эмбеддинги позволяют вычислять косинусное сходство как простое скалярное произведение.

### Поддержка внешних API {#api-embeddings}

Помимо локальных моделей, Lingtrain поддерживает вычисление эмбеддингов через внешние API:

- **Hugging Face Inference API** — любая модель SentenceTransformers, размещённая на HF
- **OpenAI Embeddings API** — модели типа `text-embedding-3-small`

Инференс через API полезен, когда нужна конкретная модель, слишком большая для локального запуска, или когда ресурсы GPU ограничены.

## Пакетная обработка {#batches}

Большие тексты разбиваются на пакеты (батчи) для обработки. Это позволяет контролировать использование памяти и обеспечивает инкрементальное выравнивание с отслеживанием прогресса.

### Как работают батчи {#how-batches-work}

Для исходного текста из `N` предложений и настроенного размера батча `B`:

1. Исходный текст делится на батчи по `B` предложений
2. Для каждого исходного батча соответствующее целевое окно рассчитывается **пропорционально**:
   - Коэффициент пропорциональности: `k = len(target) / len(source)`
   - Начало целевого окна: `source_start × k - window + shift`
   - Конец целевого окна: `source_end × k + window + shift`
3. Параметр **window** добавляет дополнительные предложения с каждой стороны для предотвращения ошибок на границах
4. Параметр **shift** позволяет вручную корректировать смещение, когда количество предложений существенно различается

### Пример {#batch-example}

Для текста с 600 исходными предложениями, 700 целевыми, размером батча 200 и окном 40:

- **Батч 1**: исходный [1–200], целевой [1–200 × 700/600 + 40] ≈ [1–273]
- **Батч 2**: исходный [201–400], целевой [201 × 700/600 - 40 ... 400 × 700/600 + 40] ≈ [195–507]
- **Батч 3**: исходный [401–600], целевой [401 × 700/600 - 40 ... 700] ≈ [428–700]

Обратите внимание на перекрытие между батчами — это намеренно. Перекрытие окон гарантирует, что предложения вблизи границ батчей не будут пропущены.

## Матрица сходства {#similarity-matrix}

Для каждого батча выравниватель строит **матрицу косинусного сходства** между эмбеддингами исходных и целевых предложений.

### Вычисление {#computation}

Для матрицы исходных эмбеддингов `S` (размерность: source_batch × embedding_dim) и матрицы целевых эмбеддингов `T` (размерность: target_window × embedding_dim):

```
similarity = S × T^T
```

Это одно матричное умножение (оптимизированное BLAS), создающее матрицу, где `similarity[i][j]` — косинусное сходство между исходным предложением `i` и целевым предложением `j`.

### Маскирование окна {#masking}

Для предотвращения ложных совпадений из отдалённых частей текста применяется **маска окна**:

```
Для каждой позиции исходного i и целевого j:
  если j × k находится в пределах [i - window, i + window]:
    сохранить similarity[i][j] (минимум 0.01)
  иначе:
    установить similarity[i][j] = 0
```

Это ограничивает совпадения диагональной полосой, обеспечивая предположение о том, что порядок предложений в целом сохраняется в переводах. Минимальное значение 0.01 предотвращает нулевое сходство в пределах допустимого окна.

## Выбор лучшего соответствия {#best-match}

После вычисления замаскированной матрицы сходства выравниватель выбирает **лучшее соответствие** для каждого исходного предложения:

```
Для каждой строки i (исходное предложение):
  best_target[i] = argmax(similarity[i])
```

Это создаёт отображение «один к одному» из исходного в целевой текст. При этом несколько исходных предложений могут отображаться на одно и то же целевое — это ожидаемо, когда переводчик объединил несколько предложений в одно.

Результирующее выравнивание сохраняется как **индекс документа**: отображение из идентификаторов батчей в списки пар `(source_ids, target_ids)`.

## Обнаружение конфликтов {#conflicts}

После выравнивания система сканирует **конфликты** — разрывы непрерывности в последовательности выравнивания.

Конфликт возникает, когда цепочка выравнивания прерывается: если исходные предложения [1, 2, 3] отображаются на целевые [5, 3, 8], порядок нарушен. Система обнаруживает эти разрывы следующим образом:

1. Проход по выровненным парам в порядке исходного текста
2. Отслеживание ожидаемой целевой последовательности (монотонно возрастающей)
3. Маркировка любой позиции, где целевая последовательность прыгает назад или пропускает

Последовательные выровненные пары без разрывов образуют **цепочки** (минимальная длина: 2–3 пары). Промежутки между цепочками являются конфликтами.

### Типы конфликтов {#conflict-types}

Конфликты характеризуются их размером — количеством вовлечённых исходных и целевых предложений:

- **Малые конфликты** (2×2, 2×3, 3×2) — наиболее частые, обычно вызваны разделением/объединением предложений
- **Средние конфликты** (до 6×6) — обрабатываются автоматическим разрешением
- **Большие конфликты** (>6×6) — отмечаются для ручного просмотра в редакторе

## Разрешение конфликтов {#resolution}

Автоматический решатель обрабатывает конфликты от наименьших к наибольшим, используя полный перебор возможных группировок.

### Алгоритм разрешения {#resolution-algorithm}

Для конфликта с `N` исходными и `M` целевыми предложениями:

1. **Перечисление всех группировок** — генерация всех возможных способов разбить `N` исходных предложений на группы и `M` целевых в соответствующие группы
2. **Вычисление групповых эмбеддингов** — для каждого варианта группировки вычисляется комбинированный эмбеддинг для каждой группы с помощью взвешенного усреднения (вес по длине предложения)
3. **Оценка вариантов** — вычисление суммы косинусных сходств между парами групп
4. **Выбор лучшего** — выбор варианта группировки с наивысшей суммарной оценкой сходства
5. **Применение** — объединение предложений согласно выбранной группировке и обновление базы данных

### Методы агрегации {#aggregation}

При объединении нескольких предложений в групповой эмбеддинг доступно несколько методов:

- **Взвешенное среднее** — вес каждого эмбеддинга пропорционален длине предложения в символах
- **Масштабирование по длине** — умножение эмбеддинга на длину, затем усреднение
- **Max pooling** — покомпонентный максимум по всем эмбеддингам предложений
- **Логарифмическое масштабирование** — вес по формуле `log(length + offset)` для убывающей отдачи на длинных предложениях

По умолчанию используется взвешенное среднее, которое придаёт больше влияния длинным (обычно более информативным) предложениям.

### Итеративная обработка {#iterative}

После разрешения раунда конфликтов система повторно проверяет оставшиеся проблемы. Разрешение одного конфликта иногда может разрешить и соседние, поскольку цепочки выравнивания растут и поглощают ранее неоднозначные области.

## Метрики качества {#metrics}

Качество выравнивания измеряется **оценкой цепочки**:

```
score = 1 - (breaks / total_lines)
```

Где:
- `breaks` = количество непоследовательных скачков в последовательности выравнивания
- `total_lines` = общее количество выровненных строк

Оценка **1.0** означает идеальную диагональ без разрывов. Оценки выше **0.95** указывают на хорошее качество выравнивания. Более низкие оценки говорят об областях, требующих ручного внимания.

## Визуализация {#visualization}

Каждый батч создаёт точечную диаграмму, где:

- **Ось X** — позиция исходного предложения
- **Ось Y** — позиция сопоставленного целевого предложения

Идеальное выравнивание даёт прямую диагональную линию. Типичные паттерны:

- **Чистая диагональ** — отличное выравнивание, мало конфликтов или их нет
- **Ступенчатая диагональ** — переводчик добавил или убрал содержимое блоками
- **Разбросанные точки** — плохое выравнивание, нужна корректировка параметров (увеличение окна, исправление сдвига или использование подстрочника)

Визуализация также опционально включает линию регрессии, показывающую ожидаемое пропорциональное отображение.

## Советы для лучших результатов {#tips}

1. **Тщательно готовьте тексты** — удалите лишнее содержимое (номера страниц, сноски), у которого нет соответствия в другом языке
2. **Используйте теги разметки** — правильные теги заголовков и авторов улучшают структурное выравнивание
3. **Начинайте с настроек по умолчанию** — размер батча 200, окно 40 хорошо работает для большинства текстов
4. **Проверяйте диагональ рано** — после первого батча проверьте визуализацию. Если она сломана, скорректируйте сдвиг перед продолжением
5. **Используйте подстрочник для редких языков** — переведите машинным переводом разбитый текст и загрузите как подстрочный перевод
6. **Разрешайте конфликты перед редактированием** — автоматическое разрешение справляется с большинством случаев; ручное редактирование — для оставшихся немногих
