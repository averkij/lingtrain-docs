# Пользовательская история: создание параллельного корпуса для исследований {#building-corpus}

Алексей Петров — исследователь в области компьютерной лингвистики. Он работает над статьёй, сравнивающей модели эмбеддингов предложений для англо-русской языковой пары, и ему нужен качественный параллельный корпус для оценки. Он решает создать его из классической литературы с помощью Lingtrain Aligner.

## Требования к исследованию {#requirements}

Алексею нужны:

- Не менее 5 000 выровненных пар предложений для статистической значимости.
- Выравнивание на уровне предложений (не абзацев).
- Чистые, верифицированные пары с минимальным шумом.
- Несколько исходных текстов для избежания доменного смещения.
- Экспорт в текстовом формате для лёгкой обработки Python-скриптами.

Он выбирает пять произведений из общественного достояния с известными русскими переводами:

1. Артур Конан Дойл — "A Study in Scarlet" / «Этюд в багровых тонах»
2. Джек Лондон — "White Fang" / «Белый клык»
3. О. Генри — "The Gift of the Magi" (сборник рассказов) / «Дары волхвов»
4. Марк Твен — "The Adventures of Tom Sawyer" / «Приключения Тома Сойера»
5. Герберт Уэллс — "The Time Machine" / «Машина времени»

## Подготовка текстов {#preparation}

Алексей скачивает английские тексты с Project Gutenberg и русские переводы с Lib.ru. Он пишет простой Python-скрипт для автоматизации очистки:

```python
import re

def clean_text(text):
    # Удаление заголовка/подвала Project Gutenberg
    start = text.find("*** START OF")
    end = text.find("*** END OF")
    if start > 0 and end > 0:
        text = text[text.find("\n", start) + 1:end]

    # Удаление номеров страниц (одиночные цифры на отдельной строке)
    text = re.sub(r'\n\d+\n', '\n', text)

    # Объединение жёстко перенесённых строк внутри абзацев
    text = re.sub(r'(?<!\n)\n(?!\n)', ' ', text)

    return text.strip()
```

Для каждой пары текстов он:

1. Очищает и нормализует текст.
2. Добавляет теги `%%%%%author.` и `%%%%%title.`.
3. Добавляет теги `%%%%%h1.` для заголовков глав.
4. Сохраняет оба файла в UTF-8 `.txt`.
5. Проверяет, что количество тегов разметки совпадает.

Подготовка занимает около часа для всех пяти текстов.

## Пакетная обработка в Lingtrain {#batch-processing}

Алексей загружает все десять файлов на вкладку Documents — пять английских и пять русских. Создаёт пять отдельных выравниваний, по одному на книгу.

Для каждого выравнивания он использует одинаковые настройки:

- **Batch size:** 200 (по умолчанию)
- **Window:** 40 (по умолчанию)
- **Batch count:** все (для обработки всего сразу)

Начинает с «Этюда в багровых тонах» — нажимает «Align all» и ждёт. Текст содержит около 800 предложений, что даёт 4 пакета. Обработка занимает несколько минут.

После завершения всех пакетов он проверяет визуализацию. Диагональ чистая для пакетов 1-3, но пакет 4 показывает дрейф — русский перевод второй части романа был реструктурирован. Он корректирует shift до +5 для пакета 4, стирает его и перезапускает выравнивание. Диагональ улучшается.

Разрешает конфликты («Resolve all»), затем быстро просматривает редактор. Выравнивание выглядит надёжно.

Алексей повторяет это для всех пяти книг. Весь процесс — загрузка, выравнивание, разрешение, проверка — занимает около 2 часов для всех пяти текстов.

## Контроль качества {#quality-control}

Для исследования качество выравнивания критично. Алексей реализует стратегию выборочной проверки:

1. Для каждой книги экспортирует **корпус сходства** для получения оценок уверенности каждой пары.
2. Случайным образом выбирает 50 пар из каждой книги (250 всего) и вручную их проверяет.
3. Рассчитывает точность выравнивания: 243 из 250 пар верны (точность 97,2%).

7 некорректных пар — в основном очень короткие предложения («Да.», «Нет.», «Конечно.»), сопоставленные с неправильным экземпляром. Для его исследования этот уровень шума приемлем, но он отмечает его в разделе методологии статьи.

Он также проверяет общий показатель цепочки (chain score), сообщаемый сводкой визуализации каждого выравнивания. Все пять выравниваний набирают выше 0,96, что подтверждает хорошее качество.

## Экспорт корпуса {#exporting}

Для каждого выравнивания Алексей переходит на вкладку Create и скачивает:

- **Корпус предложений** (исходный и целевой) — текстовые файлы с одним предложением на строку, строго выровненные по позиции.
- **TMX-корпус** — как дополнительный формат для кафедры переводоведения.
- **Базу данных выравнивания (.lt)** — для архивирования и воспроизводимости.

Теперь у него 10 текстовых файлов (5 исходных + 5 целевых), которые он объединяет в единый корпус:

```python
import os

all_en = []
all_ru = []

for book in books:
    with open(f"{book}_en.txt") as f:
        all_en.extend(f.readlines())
    with open(f"{book}_ru.txt") as f:
        all_ru.extend(f.readlines())

# Запись объединённого корпуса
with open("corpus_en.txt", "w") as f:
    f.writelines(all_en)
with open("corpus_ru.txt", "w") as f:
    f.writelines(all_ru)

print(f"Всего пар: {len(all_en)}")
# Вывод: Всего пар: 6847
```

Объединённый корпус содержит 6 847 выровненных пар предложений — значительно выше минимума в 5 000.

## Использование Python-библиотеки {#python-library}

Для более крупных проектов Алексей отмечает, что Python-пакет `lingtrain-aligner` может автоматизировать весь конвейер программно — загрузку текстов, выравнивание, разрешение конфликтов и экспорт результатов — без веб-интерфейса. Это было бы полезно для обработки десятков текстов пакетом.

Веб-интерфейс, однако, незаменим для этапа проверки качества: визуальный редактор и инструменты визуализации позволяют легко обнаруживать проблемы, которые были бы невидимы в сырых данных.

## Анализ и результаты {#analysis}

С готовым корпусом Алексей запускает сравнение моделей эмбеддингов:

1. Делит корпус на обучающую (80%) и тестовую (20%) выборки.
2. Вычисляет эмбеддинги каждой оцениваемой моделью.
3. Измеряет метрики качества выравнивания: precision, recall и F1 на уровне предложений.
4. Анализирует паттерны ошибок — какие типы предложений сложнее всего для каждой модели.

Параллельный корпус служит эталоном для оценки. Поскольку выравнивание Lingtrain было проверено автоматическим резольвером и вручную просмотрено, оно обеспечивает надёжный золотой стандарт.

Его статья публикуется через шесть месяцев с описанием методологии построения корпуса и инструмента Lingtrain Aligner.

## Распространение корпуса {#sharing}

Алексей делает корпус общедоступным (поскольку исходные тексты — из общественного достояния). Он делится:

- Сырыми файлами корпуса предложений.
- TMX-файлами для переводческого сообщества.
- Файлами баз данных `.lt` для тех, кто хочет просмотреть или изменить выравнивание.
- Файлом метаданных с документацией исходных текстов, параметров выравнивания и метрик качества.

Другие исследователи используют его корпус для своих экспериментов, и несколько работ ссылаются на методологию его статьи.

## Уроки для исследователей {#lessons}

1. **Несколько исходных текстов снижают доменное смещение.** Корпус из одного романа будет смещён в сторону стиля и словаря этого автора.
2. **Контроль качества обязателен.** Случайная выборка с ручной верификацией даёт надёжную оценку точности.
3. **Chain score — полезная прокси-метрика.** Значения выше 0,95 обычно указывают на хорошее качество выравнивания.
4. **Экспортируйте в нескольких форматах.** Корпус предложений для вычислений, TMX для переводческого сообщества, .lt для воспроизводимости.
5. **Документируйте методологию.** Записывайте параметры выравнивания, шаги разрешения конфликтов и метрики качества для воспроизводимости.
6. **Веб-редактор незаменим для проверки.** Программные конвейеры быстры, но упускают проблемы, очевидные визуально.
